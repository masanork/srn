# Web/A Folio

> **Wallet（財布）ではなく、Folio（書類鞄）。**

**Web/A Folio** は、Web/A Form エコシステムにおける「ユーザーデータの保管領域」を指す概念である。
従来の「デジタルウォレット」が持つ金融・資産管理のニュアンスを超え、個人の活動履歴、作成した文書、そしてアイデンティティを証明する鍵を統合的に管理する、ユーザー主権のデータコンテナである。

## コンセプト: データと知能の疎結合

Web/A Folio の核心は、**「データ（Folio）」と「知能（Agent）」の完全な分離**にある。

*   **The Folio (Data)**: 静的で永続的な資産。ユーザーのPCやクラウドストレージ上の「単なるフォルダー」として存在する。特定のベンダーやアプリにロックインされない。
*   **The Agent (Intelligence)**: 動的で交換可能な知能。Gemini, Claude, ChatGPT, あるいはローカルLLMなど、その時々で最適なAIを「雇い」、Folioへのアクセス権を与えて作業を依頼する。

ユーザーは Folio を持ち歩き、必要に応じて Agent を付け替えることができる。

## 例外と最適化: AIのためのインデックス

原則として Folio 内のデータは人間可読（Human-Readable）であるべきだが、ファイル数が増大した際の検索速度や、Agent のコンテキスト理解を高速化するために、**機械処理に特化したインデックスやデータベース** を保持することは、例外として許容される。

*   **Cache as Artifact**: これらはあくまでオリジナルデータ（HTML等）から生成された「キャッシュ」や「派生物」と位置付けられる。削除しても、Agent が再度HTMLをスキャンすれば再構築できるものでなければならない。
*   **Hidden by Default**: ユーザーの日常的な閲覧を妨げないよう、`.index/` や `.cache/` といった隠しディレクトリに配置することが望ましい。

## 原則: 見読性と機械可読性の両立

Web/A Form の哲学（理念）に則り、Folio に格納されるすべてのファイルは、**「人間が目で見て理解できる（見読性）」** と同時に **「Agentが処理できる（機械可読性）」** という性質を兼ね備えるべきである。

*   **ブラックボックスを作らない**: 特殊なバイナリ形式や、開発者しか読めないJSON/XML設定ファイルは極力避ける。
*   **HTML as a Container**: プロフィール情報や証明書も、ブラウザで開けば「カード」や「賞状」として閲覧できる HTML ファイルであることを基本とする。その内部に JSON-LD 等の構造化データを含めることで、Agent への可読性を担保する。

## 哲学: API連携から、本人ハブ連携へ

「システム同士をAPIで繋げば解決する」という考え方は、しばしば現実の壁に直面する。N個のサービスが相互連携しようとすれば、接続数は `N * (N-1) / 2` に膨れ上がり、契約・仕様調整・保守のコストは指数関数的に増大する（フルメッシュ接続の不可能性）。

Web/A Folio は、**「本人（ユーザー）」をデータ連携のハブ（結節点）とするアーキテクチャ** を提案する。

1.  **O(N) の接続コスト**: 全てのサービスは「本人（Folio）」に対してデータを出力し、本人からデータを受け取るだけでよい。個別のサービス間連携は不要になる。
2.  **責任と同意の復権**: API連携ではデータが裏側で自動的に流れるため、ユーザーの関与が希薄になりがちである。Folioモデルでは、データは必ず「本人」の手元を経由し、本人が中身を確認し、署名（合意）した上で相手に渡される。
3.  **AI 親和性 (AI Readiness)**: 各社独自のAPIをAIに学習させるコストは高い。しかし、「FolioにあるHTML文書を読む」というスキルは汎用的である。人間向けのインターフェース（可読文書）をそのままAIの入力とすることで、AIエージェントの導入障壁を劇的に下げることができる。

Web/A Folio は、ファイルシステム上で以下のような構造を持つことを想定している。これは Agent がユーザーの文脈を理解するための標準的なスキーマ（Context Schema）としても機能する。

```text
MyFolio/
├── .index/            # [例外] AI用の検索インデックスやベクターDB（キャッシュ）
│   ├── vectors.db
│   └── knowledge_graph.json
├── profile.html       # 基本4情報。ブラウザで見ると「名刺」や「IDカード」のように見える。
├── keys/              # 認証・署名用の鍵情報（公開鍵証明書など）。秘密鍵はハードウェアに委ねることも一般的。
├── history/           # 過去に作成・提出した Web/A Form。これらも全て閲覧可能なHTML。
│   ├── 2024-01_workcert.html
│   └── 2023-03_tax_return.html
├── certificates/      # 第三者から発行された証明書 (VC)。「賞状」や「合格証」として閲覧可能。
│   ├── degree.html
│   └── residence.html
└── inbox/             # これから記入・処理すべき空のフォーム
    └── new_application_form.html
```

## ワークフロー: Agent による代理記入

1.  **Throw in**: ユーザーは、記入が必要な Web/A Form を `inbox/` に放り込む。
2.  **Attach Agent**: ユーザーは好みの AI Agent (例: Gemini) に対し、Folio へのアクセスを許可する。
3.  **Context Retrieval**: Agent は `history/` (過去の記入履歴) や `certificates/` (証明書) を読み取り、ユーザーの文脈を理解する。
4.  **Auto-Fill**: Agent は文脈に基づき、`inbox/` 内のフォームを代理記入する。
    *   過去に類似のフォームがあれば、その書きぶりを模倣する。
    *   証明書が必要な項目であれば、`certificates/` から適切な VP を生成・添付する。
5.  **Sign & Verify**: 最後にユーザーが内容を確認し、`keys/` 内の鍵を用いて署名（PassKey認証など）を行い、完了する。

## 実装とセキュリティ

Web/A Folio の参照実装は、ユーザーのローカル環境で動作するコマンドラインツール群（CLI）である。
しかし、サードパーティーがサーバーサイドで Folio ホスティングサービスを提供することは妨げない。むしろ、高度なセキュリティ対策やバックアップの観点からは、信頼できる管理者が提供するクラウド環境の方が「本命」の解決策となるかもしれない。

ただし、Folio には個人の全生活史とも言える情報が集まるため、データが特定の一社に一極集中することは避けるべきである。ユーザーが自身の Folio をどこに置くか（ローカルか、どのクラウドか）を自由に選択・変更できるエコシステムが不可欠である。

## 本人性の確立: PassKey × マイナンバーカード

汎用的なWeb環境で、ハードウェアレベルの安全な鍵管理を行う手段は **PassKey (WebAuthn)** が事実上の標準である。しかし、PassKey 単体では「それが誰の鍵か」を第三者に証明することはできない。
そこで Web/A Folio では、日本の公的個人認証サービス（JPKI）を組み合わせた以下のモデルを採用する。

1.  **PassKey生成**: ユーザーのデバイス（スマホやPC）で PassKey ペアを生成する。
2.  **Identity Binding**: マイナンバーカード（署名用電子証明書）を用いて、生成した PassKey の公開鍵に署名を行う。
3.  **Ownership VC**: 「このPassKey公開鍵は、マイナンバーカード保有者である私が管理している」という自己署名 Verifiable Credential (VC) を作成し、Folio に格納する。
4.  **Presentation**: フォーム提出時などは、この VC を Verifiable Presentation (VP) として内包し、PassKey で署名して送信する。

これにより、日常的な操作は生体認証（Touch ID/Face ID）だけで完結しつつ、その署名はマイナンバーカードと同等の本人性を間接的に証明することが可能となる。

## なぜ "Wallet" ではないのか

*   **能動性**: Wallet は「しまう場所」だが、Folio は Agent と組み合わせることで「仕事をする場所」となる。
*   **汎用性**: 金融資産だけでなく、申請書、契約書、作品、健康診断結果など、人生のあらゆる「ドキュメント」を扱う。
*   **親和性**: ファイルベースのアーキテクチャである Web/A Form にとって、フォルダーのアナロジー（書類挟み）は最も自然である。
