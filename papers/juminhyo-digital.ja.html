検討資料：住民票の写し等の電子交付における要件定義と Web/A による実現方式

2024年12月25日
楠 正憲

1. はじめに

総務省「デジタル技術を活用した効率的・効果的な住民基本台帳事務等のあり方に関するワーキンググループ」における「中間とりまとめ」（2025年6月）では、住民票の写し等の電子交付について、利便性の向上と事務の効率化、そして原本性の確実な担保が強く求められている。

本資料では、先のホワイトペーパーで提案したアーカイブ品質のWeb文書フォーマット 「Web/A」 を前提とし、中間とりまとめで示された要件をいかに技術的に解決し、社会実装可能なものとするかについて、具体的な実現方式を提案する。

2. 中間とりまとめにおける主要要件の整理

住民票の写し等の電子交付について、中間とりまとめでは「デジタル技術を活用した住民サービスの向上」と「事務の効率化」を両立しつつ、制度的な信頼性を担保するための以下の主要な技術的・実務的要件が示されている。

  1. 実質的な真正性の担保（改ざん・偽造の防止）: 交付された電子的な証明書が、自治体によって発行されたときの内容と相違ないことが、第三者（民間窓口等）によって暗号技術を用いて客観的かつ確実に確認可能であること。
  2. 券面表示事項とデータの整合性: 住民が画面上で目視する内容（券面表示）と、システムが機械的に処理する構造化データが完全に一致しており、表示とデータの「食い違い」による誤認や不正を招かないこと。
  3. スクリーンショット等の利用による不正防止: 単純なコピーや画面キャプチャによる提示ではないことを担保し、提示者が正当な所持者であることを確認できる（なりすましを防止する）仕組みを備えること。
  4. 検証プロセスの簡便性と民間利活用の促進: 金融機関等の民間窓口において、高度な専門知識や特殊な設備を介さずとも、オンライン・オフラインを問わず迅速かつ容易に真正性が確認できること。

3. Web/A フォーマットによる技術的解決策

Web/A は、以下の技術的特徴により、上記の要件を Web 標準の延長線上で解決する。

3.1. 真正性の担保と券面・データの整合性 (HMP)

Web/A 文書は、券面表示のための HTML と、データ処理のための JSON-LD を単一のファイル内で密結合させる。

  • 整合性の署名証明: 生成ツールは、HTML コンテンツのハッシュ値と JSON-LD データを一括して署名対象に含める。これにより、PDF においてしばしば問題となる「表示内容と署名対象データの乖離（Visual Masking Attack）」を構造的に排除し、券面とデータの完全な一致を保証する。
  • ハイブリッド署名: 古典暗号（Ed25519）と耐量子暗号（ML-DSA-44）を組み合わせて署名し、将来的な量子コンピュータの脅威からも文書の真正性を長期にわたって保護する。

3.2. Passkey を活用した所持者確認の自動化 (Holder Binding)

中間とりまとめで示された「スクリーンショット等による不正防止」の鍵は、文書そのものに 「所持者の提示意思」 を動的に付与できるかにある。

  • オンデバイス署名: Web/A 文書の Viewer レイヤーに検証ロジックを内蔵する。提示の際、ユーザーがスマホの Passkey (WebAuthn) で認証することで、その瞬間限定の「提示証明（Verifiable Presentation）」をブラウザ内で生成・付与する。
  • 専用アプリ不要: この処理は標準的なモバイルブラウザ上で完結するため、住民に特定アプリのインストールを強いることなく、安全かつ簡便な所持者確認を実現できる。

3.3. 高い暗号アジリティと持続性

住民基本台帳事務は極めて長期の継続性が求められる。

  • DID (Decentralized Identifier) による信頼: DNS や WebTrust 認定の TLS 鍵を起点としつつ、DID を用いて発行鍵を管理する。これにより、将来のアルゴリズム移行（暗号アジリティ）の際も、発行者のアイデンティティを維持したまま、安全に最新の暗号方式へ「洗い替え」を行うことが可能である。

4. 住民票電子交付の運用モデル案

4.1. 発行フロー（自治体）

  1. データ生成: 基幹システムから住民票データを抽出。
  2. Web/A ビルド: 自治体内のビルドサーバー（生成ツール）が Web/A-1p 準拠の HTML+JSON-LD を生成。
  3. 発行体署名: 自治体のアイデンティティ（DID）に基づく電子署名を付与し、住民のマイナポータル等へ送付。

4.2. 提示・検証フロー（民間窓口・金融機関等）

  1. 提示: 住民がブラウザで Web/A 文書を開き、窓口でスマホ画面を提示。
  2. 検証ボタンの実行: 文書上の「真正性を確認」ボタンを押し、Passkey で認証。
  3. 真正性の証明: 画面上に「発行者：〇〇市」「整合性：正常」等の検証結果が動的に表示される。
  4. データ取得: 窓口側がデータの提出を求める場合、Web/A 文書から JSON-LD 部分を直接抽出し、基幹システムへ自動取り込みを行う。

5. 提言：今後の検討に向けた方向性

本ワーキンググループにおいては、単なる「紙のPDF化」に留まらない、「機械が処理でき、人間が視覚的に確信できる」 デジタル公文書の標準仕様を確立すべきである。Web/A フォーマットは、以下の 3 点において、電子政府の基盤として極めて有効な候補となると確信している。

  • オープンスタンダードへの立脚: 特定ベンダーの独自技術や特定のレンダリングエンジンに依存せず、ブラウザという最も普及した汎用インフラを検証環境として利用できる。
  • 長期的な情報のアクセシビリティ: セマンティック HTML とポータブル・ビューアー層を分離する多層保守モデルにより、50年、100年単位の長期保存においても閲覧性能を維持できる。
  • 次世代の信頼基盤との親和性: W3C VC や DID、耐量子暗号 (PQC) といった最新の国際標準を統合しており、将来の脅威や制度変更に対しても高い「暗号アジリティ」を有している。

本提案の方式が、効率的かつ効果的な住民基本台帳実務の実現、引いてはデジタル社会の信頼基盤の構築に寄与することを期待する。
